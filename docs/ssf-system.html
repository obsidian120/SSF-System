Operating system design doctrine

SSF System

SSF stands for Secure by default, Simple by default, and Fast by default. The SSF system defines a strict baseline for operating system design. It is not a guideline, but a decision and rejection framework.

Pillar 01

Secure by default

An OS must provide meaningful protection in its default installation, without requiring expert knowledge or post-install hardening.

Requirements

  • All applications are sandboxed by default
  • No implicit trust between applications
  • System state is protected from user-space modification
  • All software must come from verified and signed sources
  • Network exposure must be minimal and explicit

Enforcement

  • Applications receive no filesystem or network access by default
  • Permissions must be granted explicitly and scoped narrowly
  • Privilege escalation must be visible and rare
  • Any feature that bypasses isolation must be explicit, reversible, and auditable

Concrete decisions

  • No application has full home-directory access by default
  • No background service listens on the network without explicit enablement
  • No unsigned or unverifiable software is executed in trusted context
Pillar 02

Simple by default

An OS must be understandable and comfortable in normal use, with low cognitive load and minimal configuration burden.

Requirements

  • Safe defaults require no user decisions
  • The system exposes one clear mechanism per task
  • Advanced functionality must not interfere with normal usage
  • User-facing behavior must map to understandable concepts

Enforcement

  • If a feature introduces multiple ways to achieve the same goal, reject or unify it
  • If a feature requires explanation to be used safely, redesign it
  • If configuration is required for safety, reject it

Concrete decisions

  • One package manager only
  • One permissions model only
  • One service model only
  • No setup-time decisions that affect security
Pillar 03

Fast by default

An OS must feel responsive and lightweight in daily operation.

Definition of “Fast”

  • Low idle cost (CPU, memory, disk activity)
  • Low interaction latency (UI and app response)
  • Bounded background activity
  • Predictable performance under normal load

Requirements

  • No persistent background activity without clear justification
  • Services must be event-driven where possible
  • Resource usage must be measurable and explainable
  • Startup and interaction paths must be short

Enforcement

  • Any always-on service must justify why it cannot be on-demand and what its resource cost is
  • Hidden background work is not allowed
  • Features that degrade responsiveness must be visible and optional

Concrete decisions

  • No constant indexing without strict limits
  • No always-running daemons for rarely used features
  • Applications must not perform heavy initialization on launch
Core rule

Decision rule

Every major feature, package, service, and interface decision must be evaluated against SSF.

A feature must answer:

  • Does it preserve or improve default security?
  • Does it reduce or maintain simplicity?
  • Does it preserve responsiveness and efficiency?

If any answer is “no”, the feature must be redesigned or rejected.

Core rule

Priority rule

Security > Simplicity > Performance

  • Simplicity must not weaken security
  • Performance must not weaken security
  • Performance must not significantly degrade simplicity

Security

Security is non-negotiable.

Simplicity

Simplicity is enforced to prevent misuse.

Performance

Performance is enforced to preserve usability.

Core rule

Rejection rule

A feature must be rejected if it:

  • weakens security without strong justification
  • increases cognitive load unnecessarily
  • introduces persistent background cost
  • duplicates an existing mechanism
  • creates hidden system behavior

SSF is enforced through rejection. A feature is not saved by sounding useful if it breaks the baseline.

Qualification

Approval rule

A feature is SSF-approved only if it satisfies all of the following conditions.

Safety

  • Safe without configuration
  • Does not rely on user expertise

Clarity

  • Understandable in user terms
  • Does not expose internal complexity

Efficiency

  • Minimal idle cost
  • No hidden background behavior

Integration

  • Fits into existing system architecture
  • Does not introduce parallel systems
Architecture

System architecture

SSF defines concrete architectural constraints.

Kernel boundary

  • Kernel is minimal and security-focused
  • No convenience logic in privileged layers
  • Policy is handled in user space where possible

Enforced decision: Expanding kernel scope requires strong justification and review.

Application model

  • Applications are untrusted by default
  • Each app runs in a sandbox
  • Capabilities are granted explicitly

Enforced decision: No app receives broad access to filesystem, devices, or network by default.

Permissions model

  • Permissions are action-based, not global
  • Default = deny
  • Grants are contextual, limited, and revocable

Enforced decision: “Allow everything” is not a valid default.

Service model

  • Services are event-driven
  • Always-on services must justify existence
  • Service scope must be minimal

Enforced decision: Idle systems must not run unnecessary services.

Package management

  • Single unified package system
  • All packages are signed and verified
  • Installation is isolated

Enforced decision: Multiple parallel package ecosystems are not allowed.

Update model

  • Atomic updates
  • Automatic rollback on failure
  • Security updates are mandatory

Enforced decision: The system must never enter a partially updated state.

Filesystem model

  • Clear separation between system, application, and user data
  • System is protected or immutable

Enforced decision: Applications cannot modify global system state.

Networking model

  • Default deny for incoming connections
  • Per-application network permissions
  • No implicit exposure

Enforced decision: A fresh system exposes no unnecessary services.

Performance model

  • Idle system must remain lightweight
  • Background activity must be bounded
  • Resource usage must be explainable

Enforced decision: Invisible performance cost is not acceptable.

Compatibility model

  • Compatibility is secondary to SSF principles
  • Unsafe compatibility must be isolated

Enforced decision: Legacy support must not weaken default security.

Boot process

  • Boot verifies critical system integrity before handoff
  • Startup must be short, observable, and recoverable
  • Optional subsystems must not block core boot success
  • Failed boot updates return to a known-good state

Session model

  • User sessions are isolated from each other by default
  • Session startup avoids long mandatory background chains
  • User login does not imply broad trust escalation for applications
  • Ending a session revokes temporary grants and temporary runtime state

Trusted computing base

  • The trusted computing base should be kept as small as practical
  • Critical trusted components should be easy to identify and audit
  • Convenience features must not silently expand trusted code paths
  • Non-essential integrations should remain outside core trust boundaries

App runtime

  • Runtime environments should be standardized enough to avoid fragmentation
  • Applications receive only the runtime capabilities they need
  • Language runtimes and frameworks must not bypass OS security boundaries
  • Heavy runtime initialization should be minimized

Driver policy

  • Drivers are reviewed as high-risk components
  • Unsupported or legacy drivers are not granted broad trust automatically
  • Driver loading should be explicit, verified, and recoverable
  • Driver failures should be contained where architecture allows

Recovery model

  • Recovery should be available without specialist knowledge
  • The system supports rollback to a previously known-good state
  • Recovery tools should not require full manual repair of internals
  • Diagnostics should be understandable enough to guide action

Installation model

  • Installation must produce a secure baseline with no expert choices required
  • Optional features are separated from core system installation
  • First boot must not expose unnecessary services or broad permissions
  • Fresh install behavior must already meet SSF expectations
Distinction

SSF identity

SSF is distinct from existing systems.

  • Not Linux-style flexibility-first
  • Not Windows-style compatibility-first
  • Not macOS-style integration-first

SSF is default-state-first.

A system is judged primarily by what happens without configuration and what the user experiences without expertise.

Operational standard

Operational requirements

Security

  • Strong isolation by default
  • Verified software sources
  • Minimal attack surface

Simplicity

  • Usable without documentation
  • Minimal required decisions
  • Consistent system behavior

Performance

  • Responsive under normal conditions
  • Low idle resource usage
  • No unexplained background work
Practical filter

Approval checklist

  • Is it safe without extra setup?
  • Is it understandable without specialist knowledge?
  • Is its background cost minimal or justified?
  • Does it avoid duplicating an existing mechanism?
  • Does it fit the overall model cleanly?
  • Can its privilege boundaries be described clearly?
  • Can failure occur without system-wide damage?
  • Does it preserve a fast normal user experience?
Assessment

Maturity

  • Level 1 — Safe defaults exist
  • Level 2 — The system consistently follows SSF rules
  • Level 3 — Architecture is fully constrained by SSF
