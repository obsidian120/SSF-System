Versioned operating system design standard

SSF v1.0 Specification

SSF v1.0 defines a strict operating system evaluation and design standard based on three pillars: Secure by default, Simple by default, and Fast by default. It is designed to be used as a rejection framework, compliance model, and architecture guide.

Section 01

Abstract

SSF is a design and evaluation standard for operating systems. It defines what an OS must provide in its default state before users configure, harden, optimize, or customize anything.

SSF asks one primary question:

What does the system guarantee when the user does nothing?

A system may be powerful, popular, configurable, or secure in expert hands and still fail SSF. SSF approval depends on default behavior, architectural enforcement, and consistency across the system.

Section 02

Scope

SSF v1.0 applies to operating systems and OS-like platforms, including:

  • desktop operating systems
  • mobile operating systems
  • server and appliance operating systems
  • embedded operating systems
  • specialized secure operating environments

Scope declaration requirement

Every SSF evaluation must declare the target scope. A system may be SSF-approved in one scope but not another.

Valid example

SSF-approved for appliance deployments

The approval is limited, honest, and testable.

Invalid example

SSF-approved without scope

Too vague unless the system passes broadly across its full claimed use.

Section 03

Terminology

TermDefinition
Default state The behavior of the OS immediately after installation, first boot, or factory reset, before expert configuration.
SSF-approved A system or component satisfies the SSF requirements for its declared scope.
Conditionally approved A system satisfies SSF only under a defined operating mode, deployment class, or restriction.
Default-state-first The doctrine that the unmodified default experience is the primary object of evaluation.
Architecture-enforced A property is protected by system design, not merely by recommendation or user discipline.
Mechanism duplication Multiple competing ways to solve the same normal user problem without strong justification.
Section 04

Core principles

Secure by default

The OS must provide meaningful protection without post-install hardening.

Simple by default

The OS must be understandable and comfortable in normal use without expert knowledge.

Fast by default

The OS must feel responsive and lightweight without cleanup, tuning, or disabling background features.

Priority order

Security > Simplicity > Performance

  • Security must not be weakened carelessly.
  • Simplicity is preferred over unnecessary configurability.
  • Performance must remain strong enough to preserve smooth everyday use.
Section 05

Normative requirements

The following requirements define the minimum expectations for SSF v1.0. These are written as system-level requirements, not marketing claims.

Secure by default requirements

  • Applications must not receive broad trust by default.
  • System state must be protected from ordinary application modification.
  • Software authenticity must be verifiable through the default distribution path.
  • Network exposure must be minimal on first boot.
  • Privilege escalation must be visible, narrow, and auditable.
  • Security-sensitive features must fail closed where practical.

Simple by default requirements

  • Normal users must not need to understand internal architecture to use the OS safely.
  • The system should provide one primary mechanism per normal task.
  • Setup must not require expert choices to produce a safe baseline.
  • Security prompts must describe real-world effects in user terms.
  • Advanced options must not define the normal path.
  • Duplicated configuration layers must be avoided or clearly justified.

Fast by default requirements

  • Idle resource usage must be bounded and explainable.
  • Background activity must be limited, scheduled, or event-driven where possible.
  • Always-on services must justify continuous presence.
  • UI and application interaction paths must avoid unnecessary background orchestration.
  • Heavy initialization must not be hidden behind normal user actions without strong reason.
  • The system must remain responsive under normal load for its declared hardware class.
Section 06

Compliance levels

SSF v1.0 defines three compliance levels. A system must declare the highest level it satisfies.

LevelNameMeaning
Level 1 SSF Baseline Safe defaults exist, and the system avoids obvious violations such as unnecessary exposed services or unsafe setup defaults.
Level 2 SSF Consistent Major subsystems consistently follow SSF rules. Security, simplicity, and performance are visible in architecture, not only configuration.
Level 3 SSF Native The architecture is fully shaped by SSF. Exceptions are rare, explicit, contained, and documented.
Level 3 = strongest claim Conditional approval must state scope Marketing-only claims are invalid
Section 07

Approval process

An SSF approval must be based on documented evaluation, not vibes. The same criteria must be applied across all systems.

Required approval fields

  • System name and version
  • Declared scope
  • Hardware or deployment assumptions
  • Security evaluation
  • Simplicity evaluation
  • Performance evaluation
  • Exception list
  • Final verdict

Valid verdicts

VerdictMeaning
SSF-approved The system satisfies SSF for its declared scope.
Conditionally SSF-approved The system satisfies SSF only under clearly stated restrictions.
Not SSF-approved The system fails one or more pillars in its default state.
Not enough information The available evidence is insufficient for a responsible verdict.
Section 08

Test methodology

SSF testing should focus on default behavior and real user impact. Tests should not reward theoretical configurability unless it is active by default.

Default-state tests

  • Install or reset the system using normal documented steps.
  • Do not apply expert hardening.
  • Record exposed services, default software sources, update behavior, and permission behavior.
  • Test whether ordinary apps receive broad access by default.

Security tests

  • Can an application access unrelated user data without explicit permission?
  • Can an application modify system state directly?
  • Are packages, apps, or updates verified by default?
  • Are incoming network services disabled unless intentionally enabled?
  • Can the system recover from failed updates or corruption?

Simplicity tests

  • How many normal ways exist to install software?
  • How many permission models does the user encounter?
  • Does first setup ask security-critical questions?
  • Can a normal user understand why a prompt appears?
  • Are settings duplicated across multiple tools?

Performance tests

  • What runs at idle?
  • Which services are always-on, and why?
  • Is background work visible and bounded?
  • Does the system remain responsive during routine maintenance?
  • Can the system explain resource-heavy components?
Section 09

Architecture requirements

SSF compliance requires architectural support. A system cannot be considered SSF-native if the architecture contradicts the standard and only optional tools compensate for it.

Kernel and trusted core

  • The trusted computing base should be minimized.
  • Privileged code paths should remain small and auditable where practical.
  • Convenience logic should not be moved into privileged layers without strong justification.

Application model

  • Applications should be untrusted by default.
  • Capabilities should be explicit, narrow, and revocable.
  • Broad filesystem, network, device, and IPC access must not be the default.

Service model

  • Services should be event-driven where possible.
  • Long-running services must justify continuous presence.
  • Service failure should not cascade into general system instability.

Update and recovery model

  • Updates should be atomic or transactional where practical.
  • Rollback must be available for base-system failure in higher compliance levels.
  • Security updates should not require expert action.

Filesystem model

  • System data, application data, and user data must be clearly separated.
  • System state should be protected or immutable in higher compliance levels.
  • Applications must not modify global state casually.
Section 10

Exception policy

SSF does not pretend that exceptions never exist. It requires exceptions to be explicit, limited, documented, and reversible where possible.

Valid exceptions

  • A driver requiring elevated privileges for hardware support.
  • A compatibility layer for legacy software, if isolated and clearly marked.
  • An always-on service required for a core system function.
  • A broader permission grant chosen knowingly by the user for a specific reason.

Invalid exceptions

  • “Users can disable it later.”
  • “Power users understand this.”
  • “This is how legacy systems worked.”
  • “The hardware is fast enough to hide the cost.”
  • “The user clicked allow once, so broad permanent trust is fine.”
Section 11

Non-goals

SSF is intentionally not trying to maximize every possible OS value.

  • SSF is not maximum configurability.
  • SSF is not maximum legacy compatibility.
  • SSF is not maximum benchmark performance at all costs.
  • SSF is not security theater.
  • SSF is not minimalism for aesthetics alone.
  • SSF is not a brand claim without evidence.

SSF optimizes the default experience.

It values safety, clarity, and responsiveness before optional power.

Section 12

Versioning

SSF should be versioned so evaluations remain comparable over time. A system approved under one version is not automatically approved under a later version.

Version format

  • SSF 1.0 — first stable draft of the standard.
  • SSF 1.x — clarifications that do not redefine the core doctrine.
  • SSF 2.0 — major changes to approval requirements or pillar definitions.

Evaluation label format

System Name — SSF v1.0 — Verdict — Scope — Date

Example: ExampleOS 4.2 — SSF v1.0 — Conditionally approved — kiosk deployments — 2026-04-24

Scoring

SSF Score System

SSF v1.0 may be evaluated using an optional 100-point score model. The score does not replace written analysis; it standardizes comparison.

Secure — 40

App isolation, system integrity, default network exposure, update model, and privilege boundaries.

Simple — 30

Coherent system model, user cognitive load, setup complexity, and permission clarity.

Fast — 30

Idle resource usage, background services, and responsiveness.

Hard fail rules

  • If Secure is below 20/40, the system cannot be SSF-approved.
  • If Simple is below 15/30, the system cannot be fully SSF-approved for normal users.
  • If Fast is below 15/30, the system cannot be fully SSF-approved for daily general use.
Section 13

Final statement

SSF v1.0 exists to make OS design harder in the right way. It rejects systems that are safe only after hardening, simple only after learning the stack, or fast only after disabling half the platform.

A system is SSF only if security, simplicity, and speed are present in the default state, enforced by architecture, and preserved through rejection.
