Operating system design doctrine

SSF System

SSF stands for Secure by default, Simple by default, and Fast by default. The SSF system should be the core design of an OS and act as a rejection filter, design discipline, and architectural standard.

Security without post-install hardening Low cognitive load Responsive daily operation Architecture-led coherence

Pillar 01

Secure by default

An OS must provide meaningful protection in its default installation, without requiring expert knowledge or post-install hardening.

Architectural implications

  • Applications must run in isolated execution environments by default
  • User applications must not have direct write access to core system areas
  • Inter-process communication must be explicit, limited, and permission-aware
  • Services must run with the minimum privileges required
  • The default software supply chain must be verified and signed
  • The system should reduce global shared state wherever possible

Examples

  • Apps run sandboxed by default
  • System binaries and core configuration are protected from ordinary application writes
  • Background services use dedicated restricted service accounts
  • Application-to-application access is denied unless explicitly allowed
  • Software packages are signed and verified before installation
Pillar 02

Simple by default

An OS must be understandable and comfortable in normal use, with low cognitive load and minimal configuration burden.

Architectural implications

  • The system should prefer one clear mechanism over several competing ones
  • Core system behavior should be predictable and consistent
  • Security behavior should be understandable in user terms, not kernel terms
  • Configuration should be layered cleanly, with minimal overlap
  • Advanced behavior should be available, but not required for normal use

Examples

  • One primary package manager
  • One clear service model
  • One consistent permissions model
  • User-facing settings avoid exposing low-level implementation details
  • The system setup does not require the user to make expert decisions
Pillar 03

Fast by default

An OS must feel responsive and lightweight in daily operation, avoiding unnecessary background cost and interface friction.

Architectural implications

  • Services should be started on demand where possible
  • Idle resource use must be treated as a first-class design cost
  • The system should avoid heavy abstraction layers unless justified
  • Startup paths should be short and measurable
  • User interactions should not depend on unnecessary background orchestration

Examples

  • Idle CPU activity remains low
  • Memory footprint is bounded and explainable
  • Search indexing is limited, scheduled, or on-demand
  • Services are event-driven instead of permanently resident
  • Applications launch without large hidden initialization chains
Core rule

Decision rule

Every major feature, package, service, and interface decision must be evaluated against SSF.

  • Does it improve or weaken default security?
  • Does it reduce or increase user complexity?
  • Does it improve or harm responsiveness and efficiency?
  • Does it introduce a second mechanism for something already solved?
  • Does it require persistent background cost?
  • Can its value be demonstrated clearly?
Core rule

Priority rule

Security must not be weakened carelessly. Simplicity is preferred over unnecessary configurability. Performance must remain strong enough to preserve a smooth desktop experience.

Implications

  • A convenient feature should not bypass isolation casually
  • A configurable feature should not replace a better default
  • A feature that is safe and flexible but constantly heavy should be reconsidered

Test

  • If one pillar is weakened, the gain in the others must be strong and explicit
  • Trade-offs should never be hidden inside convenience language
Core rule

Rejection rule

A feature should be rejected or deferred if it significantly harms one SSF pillar without strong and demonstrable user value.

Rejection examples

  • A package format that duplicates existing system packaging without strong benefit
  • A daemon that runs permanently for a rarely used feature
  • A settings layer that overlaps with another settings layer
  • A compatibility mechanism that silently weakens isolation
  • A plugin model that allows unrestricted code injection into trusted components
Qualification

Approval rule

A feature should be considered SSF-approved only if it satisfies the following conditions.

Baseline conditions

  • Safe in its default state
  • Understandable in its default state
  • Efficient in its default state

Integration conditions

  • Fits into existing system architecture cleanly
  • Does not introduce unnecessary duplication
  • Does not require users to understand internal complexity

Operational conditions

  • Resource cost is measurable
  • Security boundaries are clear
  • Failure behavior is predictable
  • The feature can be disabled, removed, or replaced without destabilizing the system
Architecture

System architecture implications

The SSF system should directly shape the architecture of the OS.

Kernel boundary

  • The kernel should provide a small, stable, security-focused core
  • Privileged code paths should remain minimal where practical
  • Policy-heavy logic should stay in user space unless strong reasons exist otherwise
  • Drivers and privileged services should face stricter acceptance rules

User space structure

  • System services, user services, and applications should be separated cleanly
  • The OS should reduce reliance on global mutable state
  • Applications should interact through defined APIs instead of broad ambient authority

Service model

  • Services start on demand when feasible
  • Long-running services must justify continuous presence
  • Service permissions must be explicit
  • Service failures should degrade gracefully

Application model

  • Applications should be treated as untrusted by default
  • Default sandboxing and explicit capability grants
  • Limited filesystem scope and limited hardware access
  • Permissions should be revocable where possible

Permissions model

  • Permissions should be tied to actions, not broad permanent trust
  • Prompts should appear only when meaningful
  • Permission language should describe real effects
  • Persistent grants should be exceptional, not assumed

Package management

  • One primary package and update path
  • Signed repositories and clear trust boundaries
  • Security updates install automatically by default
  • Dependency complexity should be hidden from normal users

Update model

  • Atomic or transactional updates where possible
  • Reliable rollback on failure
  • Minimal need for manual recovery
  • Update interruptions should not leave the system half-configured

Filesystem design

  • Separate system state, application state, and user data
  • Protect system paths from casual modification
  • Make cleanup and recovery easier through clear structure
  • System image should be read-only or tightly protected where possible

Networking model

  • No unnecessary listening services
  • Per-application network policy support
  • Remote exposure requires explicit intent
  • Firewall policy should be enabled and sensible by default

Interface design

  • A single system settings application
  • Searchable advanced settings instead of exposing everything at once
  • Security-related prompts should use plain language
  • Animations and effects must not compromise responsiveness on normal hardware

Compatibility policy

  • Legacy support should be contained
  • Compatibility layers should be visibly distinct from native trusted paths
  • Unsafe compatibility modes should be opt-in
  • The system should prefer safe degradation over broad exceptions

Boot process

  • Boot should verify critical system integrity before handing off control
  • Startup should be short, observable, and recoverable
  • Optional subsystems should not block core boot success
  • Failed boot updates should return to a known-good state

Session model

  • User sessions should be isolated from each other by default
  • Session startup should avoid large chains of mandatory background components
  • User login should not imply broad trust escalation for applications
  • Ending a session should cleanly revoke temporary grants and temporary runtime state

Trusted computing base

  • The trusted computing base should be kept as small as practical
  • Critical trusted components should be easy to identify and audit
  • Convenience features should not silently expand trusted code paths
  • Non-essential integrations should remain outside core trust boundaries

App runtime

  • Runtime environments should be standardized enough to avoid fragmentation
  • Applications should receive only the runtime capabilities they need
  • Language runtimes and frameworks should not bypass OS security boundaries
  • Heavy runtime initialization should be minimized to preserve responsiveness

Driver policy

  • Drivers should be reviewed as high-risk components
  • Unsupported or legacy drivers should not be granted broad trust automatically
  • Driver loading should be explicit, verified, and recoverable
  • Driver failures should be contained as much as the architecture allows

Recovery model

  • Recovery should be available without specialist knowledge
  • The system should support rollback to a previously known-good state
  • Recovery tools should not require full manual repair of system internals
  • Diagnostics should be understandable enough to guide action

Installation model

  • Installation should produce a secure baseline with no expert choices required
  • Optional features should be clearly separated from core system installation
  • First boot should not expose unnecessary network services or broad permissions
  • Fresh install behavior should already meet SSF expectations
Operational standard

Operational requirements

Security

  • Verified software source path
  • Strong default isolation
  • Restricted privilege by default
  • Protected system state

Simplicity

  • Coherent system model
  • Minimal mandatory configuration
  • Clear user-facing behaviors
  • Limited mechanism duplication

Performance

  • Low idle cost
  • Responsive shell and applications
  • Bounded background activity
  • Measurable service overhead
Practical filter

SSF approval checklist

  • Is it safe without extra setup?
  • Is it understandable without specialist knowledge?
  • Is its background cost minimal or justified?
  • Does it avoid duplicating an existing mechanism?
  • Does it fit the overall model cleanly?
  • Can its privilege boundaries be described clearly?
  • Can failure occur without system-wide damage?
  • Does it preserve a fast normal user experience?
Scoring

SSF Score System

SSF v1.0 may be evaluated using an optional 100-point score model. The score does not replace written analysis; it standardizes comparison.

Secure — 40

App isolation, system integrity, default network exposure, update model, and privilege boundaries.

Simple — 30

Coherent system model, user cognitive load, setup complexity, and permission clarity.

Fast — 30

Idle resource usage, background services, and responsiveness.

Hard fail rules

  • If Secure is below 20/40, the system cannot be SSF-approved.
  • If Simple is below 15/30, the system cannot be fully SSF-approved for normal users.
  • If Fast is below 15/30, the system cannot be fully SSF-approved for daily general use.
Assessment

Maturity

  • Level 1 — Safe defaults are present
  • Level 2 — Major subsystems consistently follow SSF rules
  • Level 3 — Architecture is fully shaped by SSF, with minimal exceptions
