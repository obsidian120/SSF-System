Operating system evaluation against the SSF doctrine

SSF Evaluation List

This document evaluates real operating systems against the SSF standard: Secure by default, Simple by default, and Fast by default. The reading is intentionally strict. “Approved” is treated as a meaningful architectural verdict, not a compliment.

Method

Method and standard

This evaluation uses the stricter SSF interpretation that was developed after the earlier criticism of the concept. In this reading, an OS is not judged by whether it merely offers some security features or a clean UI. It is judged by whether its default state actually enforces the SSF baseline.

Approval thresholds used here

  • SSF-approved means the baseline model already matches SSF closely enough that approval is defensible without major excuses.
  • Conditionally SSF-approved means the system can be approved only within a narrower scope or intended operating mode.
  • Not SSF-approved means the system breaks one or more pillars in its normal default reality.

Core questions

  • Are applications meaningfully isolated by default?
  • Is the system model coherent enough for normal users to understand safely?
  • Does the system avoid unnecessary background cost and architecture sprawl?
  • Does the default state win, or does safety depend on discipline after installation?

Candidate 01

ChromeOS

Secure by default: strong Simple by default: mostly strong Fast by default: strong, but scope-sensitive Verdict: conditionally SSF-approved

Verdict: ChromeOS is the closest mainstream desktop-class candidate to an SSF pass, but only under a strict reading of its intended default model. In standard verified mode, its design philosophy lines up unusually well with SSF. Once the broader app story is treated as a core part of the platform, approval becomes conditional rather than universal.

Why it comes close

  • Its documented baseline strongly emphasizes automatic updates, sandboxing, Verified Boot, encryption, and recovery.
  • That is unusually close to SSF’s idea that users should not need post-install hardening to get meaningful protection.
  • The base system remains more unified than most mainstream desktop competitors.

Why it is not a full unrestricted pass

  • ChromeOS is cleanest when treated as a managed ChromeOS-first device.
  • Android apps, Linux container workflows, and broader flexibility layers introduce more than one application and trust model.
  • That weakens the strict SSF preference for one coherent mechanism and one coherent mental model.
Candidate 02

Ubuntu Core

Secure by default: very strong Simple by default: strong in scope Fast by default: strong in scope Verdict: SSF-approved for appliance-class systems

Verdict: Ubuntu Core is one of the clearest real-world examples of an OS that can legitimately be called SSF-approved, provided the scope is stated honestly. It maps extremely well to SSF for appliances, kiosks, embedded devices, and tightly managed deployments.

Why it passes

  • Strict confinement, transactional updates, rollback, image-like immutability, and a unified application model are strongly SSF-aligned.
  • The system reduces both repair burden and architectural sprawl.
  • Its restrictions are part of the design center rather than a bolt-on hardening layer.

Why the scope matters

  • Its strongest SSF reading is not “general-purpose desktop for everyone.”
  • It is strongest where locked-down behavior is intentional and appropriate.
Candidate 03

Fedora Silverblue / Atomic Desktop model

Secure by default: good Simple by default: insufficient Fast by default: mixed Verdict: not SSF-approved

Verdict: Fedora’s Atomic / Silverblue direction is one of the most interesting SSF-aligned architectural experiments on the desktop, but it still fails a strict approval because the total model remains too multi-layered and too conceptually split.

What it gets right

  • Immutable system direction, rollback support, default SELinux posture, and Flatpak-based sandboxing are all meaningful SSF positives.
  • It is clearly closer to SSF than traditional mutable desktop Linux.

Why it still fails

  • Users are still confronted with multiple interacting models: rpm-ostree, Flatpak, containers / Toolbx, and conventional Linux expectations.
  • That is powerful, but under SSF it is not simple by default.
  • The architecture feels transitional rather than fully resolved into one coherent everyday model.
Candidate 04

macOS

Secure by default: good to strong Simple by default: strong Fast by default: generally strong Verdict: not SSF-approved

Verdict: macOS is a strong near-miss. It has real platform protections and unusually good coherence, but it still does not enforce the kind of universal narrow application confinement that a strict SSF approval would require.

Why it scores well

  • Gatekeeper, notarization, malware protections, and the signed system volume give it a stronger baseline than many competitors.
  • Its user-facing coherence is one of the best in the mainstream desktop space.

Why it still fails

  • Not all software is confined through one strict default capability model.
  • Users can still move outside the cleanest trust path more easily than SSF would like.
  • SSF is harsher than ordinary consumer OS evaluation on exactly this point.
Candidate 05

Windows 11

Secure by default: improved, but inconsistent Simple by default: weak Fast by default: weak to mixed Verdict: not SSF-approved

Verdict: Windows 11 has made serious progress in baseline security, but it still fails SSF decisively because the broader desktop model remains broad-authority, compatibility-first, and operationally noisy.

What it gets credit for

  • Microsoft has clearly pushed hardware-backed security, VBS/HVCI, Defender, firewall defaults, and Smart App Control further than older Windows generations.
  • Some app classes benefit from capability-style privacy controls.

Why SSF still says no

  • The desktop application reality is still inconsistent; Microsoft’s own privacy docs make clear that desktop apps do not always follow the same privacy control path as modern apps.
  • The software and trust model remain fragmented: Store, MSIX, WinGet, classic installers, enterprise layers, legacy assumptions.
  • It remains fundamentally compatibility-first rather than default-state-first.
Candidate 06

Qubes OS

Secure by default: exceptional Simple by default: weak Fast by default: weak Verdict: not SSF-approved

Verdict: Qubes is a useful negative test for SSF because it proves that “extremely secure” is not enough. It passes the security pillar brilliantly, but fails the “simple” and “fast” pillars for mainstream approval.

Why it is outstanding on security

  • Its compartmentalized VM-based architecture sharply limits blast radius.
  • Template-based compartment reuse is structurally serious security engineering.

Why it still fails

  • The model is cognitively demanding for ordinary users.
  • The architecture is intentionally heavier than SSF ideally wants for normal daily operation.
Candidate 07

GrapheneOS

Secure by default: exceptional Simple by default: strong within mobile scope Fast by default: strong within mobile scope Verdict: SSF-approved for mobile scope

Verdict: GrapheneOS is one of the clearest examples of an SSF-style system in the real world, but its scope must be stated clearly: it is a hardened mobile OS for supported Pixel hardware, not a general-purpose desktop platform.

Why it passes within scope

  • It inherits Android’s modern app sandbox, verified boot, scoped permissions, and then adds further hardening beyond AOSP.
  • The trust model is coherent: supported hardware, strict baseline protections, and a narrower attack-surface philosophy.
  • The system is meaningfully secure without expecting users to become experts.

Why the scope matters

  • It is not trying to solve the same desktop openness and compatibility problem as Windows or mainstream Linux.
  • So it is a genuine SSF pass, but explicitly in the mobile category.
Candidate 08

Fedora CoreOS

Secure by default: strong Simple by default: strong in scope Fast by default: strong in scope Verdict: SSF-approved for server / appliance scope

Verdict: Fedora CoreOS deserves an SSF approval within its intended server / container-host scope. Automatic updates, atomic deployments, rollback behavior, and a controlled base system line up strongly with the doctrine.

Why it passes within scope

  • The base system is designed to be managed as a controlled image-like host rather than a loose mutable general-purpose machine.
  • Automatic updates and rollbacks reduce user burden and strengthen recovery.
  • The architecture is deliberately simpler and more disciplined than general-purpose Linux distributions.

Why scope still matters

  • This is not a mainstream personal desktop UX story.
  • Its strongest SSF reading belongs to cluster, server, and container-host roles.
Candidate 09

OpenBSD

Secure by default: culturally strong Simple by default: mixed Fast by default: decent, but not decisive Verdict: not SSF-approved

Verdict: OpenBSD deserves serious respect and is philosophically closer to SSF than many systems, but it still does not pass a hard SSF approval because its user model remains too traditional and not tightly default-confined enough for the strict doctrine.

Why it earns respect

  • OpenBSD’s long-standing security culture, privilege separation, pledge, unveil, and careful engineering make it one of the most serious security-minded Unix-like systems.
  • Its philosophy is much more disciplined than mainstream “do everything” platforms.

Why SSF still says no

  • Its security story is impressive, but it is not built around a modern one-model application sandbox / permission baseline for all normal software.
  • That matters a lot under SSF, which prioritizes default-state confinement as a first principle, not just a general culture of secure engineering.
Conclusion

Conclusion

No major general-purpose desktop OS is fully SSF-approved under a hard reading of the doctrine.

That is not a failure of the doctrine. It shows that SSF is actually demanding in a useful way.

What the evaluation reveals

  • Ubuntu Core, Fedora CoreOS, and GrapheneOS show that SSF can already exist in real systems when scope is tight and the platform is allowed to be disciplined.
  • ChromeOS shows how close a mainstream user-facing system can get when the vendor strongly unifies the baseline.
  • macOS shows that polish and serious platform security still are not the same thing as universal narrow default confinement.
  • Windows 11 shows how legacy compatibility and software-model sprawl conflict directly with SSF coherence.
  • Qubes OS proves that extreme security alone does not satisfy a three-pillar doctrine.
  • OpenBSD shows that strong security culture is not identical to strict default-state-first application control.
